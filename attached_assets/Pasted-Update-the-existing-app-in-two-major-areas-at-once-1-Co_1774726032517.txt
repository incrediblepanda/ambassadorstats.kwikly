PART 1: LIVE AMBASSADOR BACKEND API INTEGRATION
==================================================

Important integration rules:
- Use the Ambassador Backend API only
- Do not use browser snippets, frontend widget integrations, or client-side Ambassador embed logic as the data integration layer
- All Ambassador data syncing must happen server-side
- Read credentials only from Replit Secrets / environment variables
- Never hardcode credentials anywhere in code
- Never expose the API token to the browser

Add support for these environment variables / Replit Secrets:
- GETAMBASSADOR_API_BASE_URL
- GETAMBASSADOR_API_USERNAME
- GETAMBASSADOR_API_TOKEN
- APP_BASE_URL
- SESSION_SECRET
- ADMIN_BOOTSTRAP_EMAIL
- ADMIN_BOOTSTRAP_PASSWORD

Requirements for environment configuration:
1. Add startup validation for required environment variables
2. If any required secret is missing, fail gracefully with a clear message in logs
3. Add a settings/integrations page in the admin UI that shows:
   - whether GETAMBASSADOR_API_BASE_URL is configured
   - whether GETAMBASSADOR_API_USERNAME is configured
   - whether GETAMBASSADOR_API_TOKEN is configured
   - whether APP_BASE_URL is configured
   - whether the last Ambassador API connection test succeeded
   - last successful sync timestamp
4. Add a “Test Connection” action that makes a safe server-side request to the Ambassador Backend API and reports success/failure in the admin UI
5. Never display the raw API token in the UI

Build a clean Ambassador API client/service layer:
- Centralize all Ambassador API calls in one service module
- Add proper request handling, error handling, and logging
- Keep raw fetch logic separate from normalization logic
- Make this service easy to extend later

Create or improve service methods such as:
- getAllAmbassadors(page?)
- getAmbassadorByEmailOrUid(...)
- getAmbassadorStats(...)
- getCommissions(...)
- getShortcode(...)
- normalizeAmbassador(...)
- normalizeReferral(...)

Sync requirements:
- Replace mock Ambassador reads with real server-side API calls where possible
- Keep mock fallback only where the live response mapping is still unresolved
- Add raw API payload capture/storage for debugging
- Make sync idempotent
- Prevent duplicate ambassador creation
- Prevent duplicate referral creation
- Upsert ambassadors using the best durable unique Ambassador identifier
- Upsert referrals using the best available unique source identifier
- Store sync timestamps, sync outcomes, and errors
- Make manual sync safe to rerun

Auto-create synced ambassador dashboard records:
When a new ambassador is discovered during sync:
- automatically create their synced ambassador record in the internal database
- automatically generate:
  - dashboard slug
  - secure dashboard token
  - iframe URL
- persist all of those values
- mark dashboardAccountCreated = true
- surface this clearly in the admin UI

Preserve current business rules:
- Ambassador is the source of truth
- HubSpot is upstream only and should not be queried directly by this app for dashboard rendering
- Admin-only platform
- No ambassador login
- Public iframe dashboard route is secured by short code + token
- Short code alone is never enough for access

Maintain or improve the public iframe dashboard:
Route format:
- /dashboard/:shortCode?token=...

Requirements:
- validate short code + token on every request
- return unauthorized if invalid
- be iframe-friendly
- show only that ambassador’s data
- no unnecessary admin chrome on this route

Business mapping configuration:
Create or preserve a dedicated config file that maps Ambassador custom fields into business meanings:
- custom1 = companyName
- custom2 = totalShiftsWorked
- custom3 = phoneNumber
- custom4 = approvedAt
- custom5 = numberShiftsWorked
